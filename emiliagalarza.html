EMILIA GALARZA


PROCESO ACS
  • Definición de Requerimientos:

    Establecimiento de los criterios que el software debe cumplir, tanto en funcionalidades como en restricciones operativas. Comprende la recolección detallada de necesidades del usuario y la transcripción a especificaciones técnicas.

  • Diseño del Sistema:

    Creación de la arquitectura del software que incluye la definición de módulos, interfaces y datos que serán controlados y gestionados por el sistema. Se elaboran diagramas y modelos que sirven como planos para el desarrollo.

  • Desarrollo:

    Implementación de los componentes de software según el diseño, incluyendo las funciones de control y gestión de la aplicación. Codificación y configuración se llevan a cabo en esta etapa.

  • Pruebas:

    Verificación de que el sistema de control de aplicaciones cumple con los requerimientos especificados y gestiona eficientemente las operaciones de la aplicación. Se incluyen pruebas unitarias, integración, sistema y aceptación del usuario.

  • Implementación y Mantenimiento:

    Despliegue del software y monitoreo continuo para asegurar el control operativo, realizar ajustes y actualizaciones según sea necesario. Incluye la corrección de errores y la actualización de funcionalidades.


FLUJO DE TRABAJO DE GITFLOW
  • Rama Main o Master:

    Representa el código de producción, el cual está en vivo y es utilizado por los usuarios finales. Debe ser estable en todo momento, reflejando una versión fiable del software. Las actualizaciones en esta rama se realizan a través de ramas release o hotfix. Cada commit en master se etiqueta con un número de versión para el seguimiento de lanzamientos. Es la rama más crítica, ya que cualquier error aquí afecta directamente a los usuarios.

  • Rama develop:

    Sirve como rama de integración para características y mejoras nuevas. Contiene el estado más avanzado del desarrollo, pero aún no listo para producción. Todas las ramas feature se fusionan aquí una vez completadas. Actúa como base para crear ramas release y hotfix. Representa el próximo lanzamiento en preparación, con un estado menos estable que master.

  • Rama feature:

    Son ramas temporales creadas para desarrollar nuevas funcionalidades. Se bifurcan de develop y se fusionan de vuelta en ella una vez finalizada la característica. Cada rama feature se enfoca en una única funcionalidad o mejora, permitiendo a los equipos trabajar en diferentes características simultáneamente sin perturbar la rama principal de desarrollo. Se nombran de manera que reflejen la funcionalidad o tarea en desarrollo, como feature/login o feature/payment-integration.

  • Rama hotfix:

    Se utilizan para reparaciones urgentes en la versión de producción. Se bifurcan directamente de master y se destinan a corregir errores críticos encontrados en producción. Una vez completado el arreglo, se fusionan en master y develop para asegurar que la corrección esté presente en ambas ramas. Permiten una rápida respuesta a problemas sin perturbar el flujo de trabajo de desarrollo regular. Los hotfix suelen estar asociados con una versión, por ejemplo, hotfix/1.3.1.

  • Rama release:

    Preparan el próximo lanzamiento de producción, iniciando desde la rama develop. Permiten realizar pruebas finales y correcciones menores sin afectar el desarrollo continuo en develop. Una vez que la rama release está lista, se fusiona en master y de vuelta en develop. Aseguran una transición suave y controlada de las características de develop a master. Se etiquetan con un número de versión, como release/1.3.0.


FLUJO DE PROCESO GESTIÓN DE CAMBIO
  • Solicitud de Cambio (Request for Change - RFC):

    Inicio del proceso mediante la presentación de una solicitud de cambio detallada, la cual debe incluir la naturaleza, el propósito y el impacto del cambio propuesto.

  • Evaluación del Cambio:

    Análisis del cambio propuesto por un comité de control de cambios o un responsable para valorar su impacto, coste, beneficios y riesgos asociados.

  • Aprobación del Cambio:

    Decisión sobre si proceder con el cambio. Si se aprueba, se planifica y se asignan los recursos necesarios. Si se rechaza, se documentan las razones y se comunica a las partes interesadas.

  • Planificación del Cambio:

    Desarrollo de un plan detallado para implementar el cambio, que debe incluir tareas, recursos, cronograma y procedimientos de prueba.

  • Implementación del Cambio:

    Ejecución del plan de cambio bajo supervisión, con una comunicación constante a las partes interesadas sobre el progreso.

  • Pruebas y Validación:

    Pruebas exhaustivas para verificar que el cambio funciona como se esperaba y no introduce nuevos problemas en el sistema.

  • Documentación:

    Actualización de la documentación del sistema para reflejar el cambio implementado, incluyendo manuales técnicos y ayuda para usuarios.

  • Revisión Post-Implementación:

    Evaluación del cambio una vez implementado para asegurar que se han alcanzado los objetivos y que no hay efectos secundarios no previstos.
